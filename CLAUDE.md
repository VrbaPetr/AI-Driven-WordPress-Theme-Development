# AI-Driven Boilerplate — WordPress Theme

WordPress boilerplate theme for IT companies / freelancers. Classic PHP theme with ACF, Gutenberg blocks backed by ACF fields, Tailwind CSS, Alpine.js, and Vite.

## Tech Stack

| Layer | Choice |
|---|---|
| PHP | 8.2+ |
| CSS | Tailwind CSS v4 |
| JS | Vanilla JS + Alpine.js |
| Build | Vite |
| CMS fields | Advanced Custom Fields (ACF Pro) |
| Editor | Gutenberg blocks with ACF field backing |
| Linting | ESLint (JS), PHPCS with WordPress coding standards (PHP) |

## Development Commands

All commands are run via `make`. Run `make help` to see all available targets.

```bash
make install     # Install all dependencies (pnpm + composer)
make update      # Update dependencies within declared version ranges
make watch       # Start the Vite dev server with live-reload
make build       # Build production assets via Vite
make zip         # Build and package a deployable theme zip into ./dist
make check       # Run all linters (PHPCS)
make fix         # Auto-fix all linting issues (PHPCBF)
```

## Conventions

Load the relevant skill **before** writing code in that area:

| Skill | When to invoke |
|---|---|
| `/acf-field-reference` | Before reading or rendering any ACF field in a template |
| `/php-block-conventions` | Before writing or editing a block template in `template-parts/blocks/` |
| `/php-component-conventions` | Before writing or editing a component in `template-parts/components/` |
| `/php-data-conventions` | Before writing functions in `inc/functions-data.php` |
| `/php-ajax-conventions` | Before writing handlers in `inc/functions-ajax.php` |
| `/tailwind-conventions` | Before writing any CSS file in `src/css/` |
| `/js-conventions` | Before writing any JS file in `src/js/` |

For ACF blocks: use `/register-acf-block` to scaffold a new block and `/build-acf-block` to implement it.

## ACF Conventions

- All field groups, options pages, and block field definitions are **written directly as JSON files** into `acf-json/` — never created via the WP admin UI, never written in PHP code.
- Each JSON file follows the ACF Local JSON schema: a `group_*` key for the group, `field_*` keys for every field, deterministic keys derived from the group/field name.
- The `acf-json/` directory is version-controlled — JSON files are the source of truth.
- After generating or editing a JSON file, reload the field group in WP admin (ACF → Field Groups → "Sync available") to confirm it loads correctly.
- **Always use a Repeater field for any content that can appear 1 to N times** — never model repeatable items as numbered individual fields (e.g. `link_1`, `link_2`). A Repeater with a single sub-field is still correct; it keeps the data structure uniform and editor-friendly.
- Access options page fields with `get_field( 'field_name', 'option' )`.

## PHP Conventions

- Follow WordPress coding standards (PHPCS ruleset in `phpcs.xml`).
- Always include the text domain in translation functions: `__( 'Text', 'textdomain' )`.
- Keep templates free of business logic — move it to `inc/` and call from the template.
- Always escape all output.

### File docblock

Every PHP file must open with a file-level docblock:

| Location | Format | Example |
|---|---|---|
| Root templates (`page.php`, `single.php`, …) | Plain description | `The template for displaying all pages.` |
| `inc/` files | Plain description | `Data-related functions and custom queries.` |
| `template-parts/pages/` | `Page: {Name}` | `Page: Testimonial` |
| `template-parts/layout/` | `Layout: {Name}` | `Layout: Header` |
| `template-parts/forms/` | `Form: {Name}` | `Form: Contact` |
| `template-parts/components/` | `Component: {Name}` | `Component: Button` |
| `template-parts/blocks/` | `Block: {Name}` | `Block: Hero` |

```php
<?php
/**
 * Page: Testimonial
 *
 * @package {textdomain}
 */
```

### Image sizes

Custom image sizes are registered in `inc/functions-design.php` via `add_image_size()`. Use kebab-case names that describe the context. When calling `wp_get_attachment_image()`, always use the most appropriate registered size — never default to `'full'` unless the image truly needs full resolution.

## Vite + WordPress Integration

Source files live in `src/` — Vite compiles them to `assets/css/` and `assets/js/`. The Vite manifest (`.vite/manifest.json` inside `assets/`) maps entry point paths to their versioned output filenames. Assets are enqueued in `functions.php` by reading the manifest.

## Git Workflow

### Branch structure

| Branch | Purpose |
|---|---|
| `main` | Production-ready code — only updated via PR from `develop` for releases |
| `develop` | Integration branch — base for all feature/fix branches |
| `feat/*`, `fix/*`, `chore/*` … | Short-lived branches, one per task |

### Starting a task

1. Confirm the working branch is `develop` (or switch to it)
2. Pull the latest: `git pull origin develop`
3. Create a new branch: `git checkout -b feat/block-name`

**Branch naming:** use the commit type as prefix + short kebab-case description: `feat/hero-block`, `fix/mobile-nav-outside-click`, `chore/acf-json-sync`.

### Finishing a task

1. Make a final commit with any remaining changes
2. Push the branch and open a PR into `develop` using `gh pr create`
3. PR title follows the same conventional commit format as the commits inside it
4. **Never merge the PR** — the user reviews and merges manually

### My behaviour (Claude)

- **Before starting any implementation** — check the current branch. If it is `main` or `develop`, stop and ask the user to create a feature branch first.
- **Branch name not provided** — suggest a name following the `type/short-description` convention and ask the user to confirm before creating it.
- **Leave PR merging to the user** — open the PR and stop there.

## Commit Conventions

Follow [Conventional Commits](https://www.conventionalcommits.org/): `<type>(<scope>): <subject>`

- **Subject** — imperative mood, lowercase, no period, max 72 characters

| Type | When to use |
|---|---|
| `feat` | New block, component, CPT, taxonomy, shortcode, page template, or any new functionality |
| `fix` | Bug fix — broken markup, wrong field, JS error, CSS regression |
| `style` | Visual / CSS-only change with no logic change |
| `refactor` | Code restructured without changing behaviour |
| `chore` | ACF JSON sync, dependency updates, config file changes |
| `build` | Makefile, `package.json`, Vite, or PHPCS setup changes |
| `docs` | `CLAUDE.md`, `README.md`, or inline documentation only |
| `revert` | Reverting a previous commit |

## Out of Scope

- Page builders (Elementor, WPBakery)
- Full Site Editing (FSE) / block themes
- Headless / REST-only setups

## Key Decisions

- **ACF fields as generated JSON** — field group definitions written directly into `acf-json/` by Claude; no admin UI step required; version-controlled.
- **Tailwind v4 CSS-first config** — no `tailwind.config.js`; tokens live in `src/css/variables/`.
- **Always escape all output** — no exceptions, even for values that appear safe.
